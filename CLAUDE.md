# CLAUDE.md — writing-tools-nlp

## Project purpose

Portfolio project targeting Apple's Software Engineer in NLP and ML role (200666537-3337).
The Apple JD names three specific techniques: **adapters**, **speculative decoding**, and
**guided generation**. This project implements all three in one coherent application that
mirrors Apple Intelligence Writing Tools (summarise, rewrite, smart reply) — plus a fourth
Apple-alignment signal: **on-device inference via Apple MLX** on Apple Silicon.

**Target resume line:**
> Fine-tuned Qwen3-4B with QLoRA adapters for summarisation, rewriting, and smart-reply
> generation; implemented speculative decoding (0.6B draft model) achieving ~2× speedup,
> validated against HuggingFace assisted generation; deployed schema-constrained guided
> generation, with on-device 4-bit inference via Apple MLX on Apple Silicon (M1).

## Tech stack

| Component | Library | Version |
|---|---|---|
| Base model (verify / fine-tune target) | Qwen/Qwen3-4B | — |
| Draft model (speculative decoding) | Qwen/Qwen3-0.6B | — |
| Fine-tuning | peft (LoRA), trl (SFTTrainer) | latest |
| Training | transformers, torch, bitsandbytes | latest |
| Evaluation | evaluate (rouge, bertscore) | latest |
| Guided gen | outlines | latest |
| On-device inference | mlx, mlx-lm | latest |
| Demo | gradio | latest |
| Data | datasets (HuggingFace) | latest |
| Experiment tracking | optional: wandb | latest |

**Why Qwen3 (not Phi-3):** speculative decoding requires draft and verify models to share
a tokenizer — the accept/reject step compares probabilities over the same token IDs.
Qwen3 ships 0.6B → 4B with one shared tokenizer. Phi has no tiny sibling, so a
distilgpt2-draft setup would be mathematically invalid. Qwen3 is also Apache 2.0 and
currently the most widely used small open-model family.

**Hardware split:**
- Training: **Kaggle GPU** (primary — ~9h sessions, 30 GPU h/week) or Colab T4 with Drive resume. QLoRA 4-bit. Never train locally on Mac.
- On-device inference benchmark: MacBook Air M1 8GB via MLX (4-bit, inference only).
- Demo hosting: HuggingFace Spaces (see deployment notes — CPU is the constraint).

Expected training time: ~6–8 hours for 1 epoch on ~4k examples at max_seq_len=1536 (4B on T4/P100). Mid-training eval is disabled; run `train/eval.py` post-training.

## Key commands

### Setup
```bash
pip install -r requirements.txt
```

### Prepare data
```bash
python data/prepare.py
# writes data/train.jsonl and data/val.jsonl
# sources: CNN/DailyMail (summarisation) + email subset (smart reply) + news (rewrite)
# ~4k total examples, truncated to fit max_seq_len=1536
```

### Train
```bash
# Kaggle GPU (primary) — open notebooks/train_kaggle.ipynb
# Colab fallback — notebooks/train_colab.ipynb (mount Drive for checkpoint resume)
# (train/train.py is the source of truth; notebooks are thin wrappers)
python train/train.py --resume_from_checkpoint ./checkpoints/checkpoint-100   # if session died
```

### Evaluate
```bash
python train/eval.py --adapter_path ./adapter --split val
# prints ROUGE-1, ROUGE-2, ROUGE-L, BERTScore-F1 for base vs fine-tuned
# BERTScore on a 200-example subset (it's heavy)
```

### Run inference
```bash
python inference/pipeline.py \
  --task summarise \
  --input "Apple today announced..." \
  --adapter_path ./adapter
```

### Speculative decoding benchmark
```bash
python benchmarks/latency.py --adapter_path ./adapter --n_samples 50
# prints: mean latency (ms), tokens/sec, speedup ratio for:
#   1. standard generation
#   2. custom speculative decoding (this repo)
#   3. HF assisted generation (model.generate(assistant_model=...)) — reference check
```

### MLX on-device (run on the MacBook, not Colab)
```bash
python mlx_tools/convert.py --adapter_path ./adapter        # fuse + convert to 4-bit MLX
python benchmarks/mlx_bench.py                        # tokens/sec on M1
```

### Run demo locally

## Architecture

### 1. Fine-tuning (QLoRA — the "adapter")

```
Qwen3-4B-Instruct  (frozen base, 4-bit NF4 via bitsandbytes)
        +
LoRA adapter  (r=16, alpha=32, target_modules=["q_proj", "k_proj", "v_proj", "o_proj"])
        =
Fine-tuned model for 3 writing tasks
```

`train/config.py` holds all hyperparameters. Do not hardcode them in `train.py`.
Set `max_seq_len=1536` — CNN/DailyMail articles are long; truncate inputs in
`prepare.py`, not at train time. ~4k examples is enough for SFT style adaptation;
more data on a free T4 risks session timeout before convergence.

Prompt format: use the **Qwen3 chat template** via `tokenizer.apply_chat_template` —
never hand-roll the template string.

Tasks and their system instructions are defined in `data/prepare.py` as a dict
`TASK_INSTRUCTIONS`. Keep this as the single source of truth — `app.py` imports from it.

### 2. Speculative decoding (`inference/speculative_decode.py`)

Core idea: small draft model generates K tokens autoregressively; the large verify model
checks all K in one parallel forward pass; tokens are accepted/rejected by probability
ratio (rejection sampling), guaranteeing the output distribution matches the verify model.

```
draft_model  = Qwen3-0.6B           (fast — generates candidates)
verify_model = fine-tuned Qwen3-4B  (slow, accurate — accepts/rejects)
K            = 5                    (tokens per speculative step, tunable 4–8)
```

**Shared tokenizer is a hard requirement** — both models use the Qwen3 tokenizer, so
draft token IDs map directly onto verify-model logits. No vocabulary alignment hacks.

Key functions to implement:
```python
def draft_generate(model, input_ids, K: int) -> tuple[Tensor, Tensor]:
    """Returns (candidate_ids, candidate_log_probs) of shape (K,)"""

def speculative_decode(
    draft_model, verify_model, tokenizer,
    prompt: str,
    max_new_tokens: int = 100,
    K: int = 5,
) -> str:
    """Full speculative decoding loop. Returns generated text."""
```

**Validation requirement:** `benchmarks/latency.py` must also run HuggingFace's built-in
assisted generation (`verify_model.generate(assistant_model=draft_model)`) as a reference.
The custom loop's speedup should be in the same ballpark; if it isn't, the custom loop
has a bug. Report both in the README — verifying against the reference implementation is
itself an engineering signal.

Benchmark must report:
- Mean latency per call (ms): standard vs custom speculative vs HF assisted
- Tokens per second
- Acceptance rate of draft tokens
- Speedup ratio (target: 1.5–2.5×)

This is the most technically differentiated part of the project. A correct, well-measured
benchmark table matters more than clever code.

### 3. Guided generation (`inference/guided_gen.py`)

Used only for the Smart Reply task. Constrains output to valid JSON using `outlines`.

Output schema:
```python
from pydantic import BaseModel
from typing import Literal

class SmartReply(BaseModel):
    subject: str
    body: str
    tone: Literal["formal", "casual", "friendly"]
```

`outlines.generate.json(model, SmartReply)` handles the constrained sampling.
`outlines` works cleanly with Qwen3 — no fallback path needed.

### 4. On-device inference with Apple MLX (`mlx_tools/`)

The Apple-alignment centrepiece. After training:

1. Fuse the LoRA adapter into the base model (`peft` merge_and_unload, or `mlx_lm.fuse`).
2. Convert to MLX 4-bit: `mlx_lm.convert --hf-path <fused> -q --q-bits 4`.
3. Benchmark on the M1: `benchmarks/mlx_bench.py` reports tokens/sec and peak memory.

Constraints: M1 Air has 8GB unified memory. The 4-bit 4B model is ~2.5GB — inference is
fine, training is not. Run the benchmark with other apps closed. Report the M1 tokens/sec
in the README; this is the honest "on-device" latency number, and it maps directly onto
Apple Intelligence's ~3B on-device model story.

### 5. Gradio app (`app/app.py`)

Three tabs, one model load at startup:
- **Summarise** — text input → paragraph summary
- **Rewrite** — text input + tone selector (formal/casual) → rewritten text
- **Smart Reply** — email input → JSON card showing subject/body/tone

The model loads once on startup with `pipeline.py`. Do not reload per request.

**Spaces deployment reality:** free Spaces is CPU-only and a 4B transformer on CPU is
~1–3 tokens/sec — too slow for a good demo. In order of preference:
1. Export the fused model to **GGUF** and serve with llama.cpp on Spaces (meaningfully
   faster than transformers-on-CPU).
2. Apply for a **ZeroGPU** Space / community GPU grant.
3. If both fail, demo the fine-tuned **Qwen3-1.7B** variant on Spaces and state in the
   README that headline numbers come from the 4B model.

The README must say that latency numbers come from the T4 (speculative) and M1 (MLX)
benchmarks — never from the Spaces demo.

## Development notes

- **Do not commit adapter weights** — add `adapter/`, `fused/`, `mlx_model/` to
  `.gitignore`. Push to HF Hub only.
- **Do not commit data files** — `data/*.jsonl` are generated locally. Add to
  `.gitignore`. Keep `data/prepare.py` as the reproducible source.
- **GPU session limits** — Kaggle free tier: ~9h/session, 30 GPU h/week (preferred for 4B).
  Colab free tier: ~2–3h/session; use Google Drive checkpoint symlink + resume.
  SFTTrainer saves checkpoints every 100 steps; resume with
  `--resume_from_checkpoint ./checkpoints/checkpoint-XXX`.
  Mid-training eval is disabled (`eval_strategy="no"`) — it OOMs on T4 at max_seq_len=1536;
  run `train/eval.py` after training instead.
- **Qwen3 thinking mode** — Qwen3 instruct models support a thinking mode; disable it
  for these tasks (`enable_thinking=False` in `apply_chat_template`) or outputs will
  contain reasoning traces that wreck ROUGE scores and the Smart Reply JSON.
- **Never load two 4B models on the M1** — speculative decoding benchmarks (draft + verify
  in memory simultaneously) run on Colab T4 only. The M1 runs the single fused MLX model.
- **bitsandbytes is CUDA-only** — any script importing it must never run on the Mac.
  Keep MLX code paths fully separate (`mlx_tools/` and `benchmarks/mlx_bench.py`).
- **Spaces requirements.txt** — keep `app/requirements.txt` minimal and pinned; Spaces
  build failures from unpinned transformers/outlines versions are the most common
  deployment time-sink.
